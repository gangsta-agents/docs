# Contract: CI/CD Pipeline for Gangsta Site

## Objective
Add a fully automated CI/CD pipeline that builds and publishes the gangsta-site documentation site to Cloudflare Pages on every push to the `master` branch, eliminating the need for manual deployment.

## Requirements

### Functional Requirements

1. **FR-001** — The pipeline MUST trigger on every push to the `master` branch of `github.com/kucherenko/gangsta-site`
2. **FR-002** — The pipeline MUST check out the repository source code
3. **FR-003** — The pipeline MUST install all npm dependencies using `npm ci` against `package-lock.json`, with Node.js version read from `.nvmrc` (currently `20`)
4. **FR-004** — The pipeline MUST run `npm run sync` (which invokes `scripts/sync-skills.ts`) to fetch skill content from the public `kucherenko/gangsta` repo, using the automatically-provided `GITHUB_TOKEN` environment variable; the sync script requires `GITHUB_TOKEN` and exits with code 1 if absent
5. **FR-005** — The pipeline MUST run `nuxt generate` (via `npx nuxt generate`) to produce the static site output in `.output/public`
6. **FR-006** — The pipeline MUST verify the build output before deployment: (a) `.output/public` directory must exist, (b) `.output/public/index.html` must be present as the site entry point, (c) at least 20 files must exist in `.output/public` — this threshold accounts for the minimum expected static shell (HTML, JS chunks, CSS, fonts, manifests) for a functional Docus site, providing a safety net against total build failure without false-positive risk for normal builds
7. **FR-007** — The pipeline MUST deploy the contents of `.output/public` to the Cloudflare Pages project named `gangsta`, targeting branch `master` (production), using credentials from two GitHub repository secrets: **`CLOUDFLARE_API_TOKEN`** and **`CLOUDFLARE_ACCOUNT_ID`**
8. **FR-008** — The file `wrangler.toml` MUST have `pages_build_output_dir` changed from `"dist"` to `".output/public"` in the same commit as the workflow file, as a prerequisite correction; the workflow deploy command passes the output path explicitly and is not blocked by this fix, but the fix is required for consistency

### Non-Functional Requirements

1. **NFR-001 Security** — All GitHub Actions action references MUST be pinned to full commit SHAs (not floating semver tags such as `@v3` or `@v4`) to prevent supply-chain attacks; this includes `actions/checkout`, `actions/setup-node`, and `cloudflare/wrangler-action`
2. **NFR-002 Security** — The workflow MUST declare a minimal `permissions` block with only `contents: read` (required by `actions/checkout`); `deployments: write` is NOT required as deployment occurs via the Cloudflare API token, not the GitHub Deployments API
3. **NFR-003 Reliability** — The workflow MUST include a `concurrency` block with `group: deploy-production` and `cancel-in-progress: false`; an in-flight deploy must never be cancelled by a subsequent push
4. **NFR-004 Performance** — The `actions/setup-node` step MUST use `cache: 'npm'` to cache the npm content-addressable store at `~/.npm`, reducing install time on repeat runs
5. **NFR-005 Correctness** — The cache target MUST be `~/.npm` (npm tarball cache), NOT `node_modules`; `npm ci` unconditionally deletes `node_modules` before installing, making a `node_modules` cache non-functional
6. **NFR-006 Secrets** — `CLOUDFLARE_API_TOKEN` and `CLOUDFLARE_ACCOUNT_ID` MUST be stored as GitHub repository secrets; they MUST NOT appear in any committed file

## Grilling Conclusions

### Key Decisions
- SHA-pinned action refs: eliminates supply-chain attack surface on the deployment token path
- Concurrency group with `cancel-in-progress: false`: prevents race condition where two pushes produce overlapping deploys; the losing deploy (last-write-wins) scenario is worse than queuing
- `cache: 'npm'` via `setup-node`: correct cache strategy for `npm ci` — caches tarballs not compiled binaries
- Post-build verification before deploy: catches silent `nuxt generate` partial failures at zero cost
- `wrangler.toml` fix in same PR: aligns config with actual Nuxt Cloudflare Pages output path; should not be deferred
- `deployments:write` removed: deploy uses CF API token, not GitHub Deployments API

### Rejected Alternatives
- **PR preview deployments**: Don explicitly declined; master-only is the specified scope
- **Caching `node_modules` instead of `~/.npm`**: Anti-pattern with `npm ci` — npm ci deletes it unconditionally
- **Floating action semver tags**: Rejected on security grounds; any action receiving secrets must be SHA-pinned
- **`cancel-in-progress: true`**: Rejected — cancelling an in-flight deploy risks serving a half-deployed state; queuing is safer

### Unresolved Objections (Accepted Risks)
- **Live GitHub API dependency in sync path** — Risk: MEDIUM — Mitigation: sync script exits 1 on API error, blocking deploy rather than deploying empty content
- **Non-atomic sync** — Risk: LOW — Mitigation: same as above; any API error during sequential file fetch aborts the build
- **Hardcoded "20 skills" in sync-generated index.md** — Risk: LOW — separate heist

## Acceptance Criteria

1. A file `.github/workflows/deploy.yml` exists and is syntactically valid GitHub Actions YAML
2. A push to `master` triggers the workflow run in the repository's Actions tab and completes without error
3. The deployed site is accessible at the Cloudflare Pages URL with content reflecting the pushed commit
4. `wrangler.toml` shows `pages_build_output_dir = ".output/public"`
5. The workflow correctly fails (exits non-zero, does not proceed to deploy) when build output verification fails
6. No values for `CLOUDFLARE_API_TOKEN` or `CLOUDFLARE_ACCOUNT_ID` appear in any committed file; they are documented as required repository secrets
7. The `GITHUB_TOKEN` required by the sync step is the auto-provided Actions token (no PAT required)

## Out of Scope

- PR preview deployments (explicitly declined by Don)
- Fixing the `skills/.well-known` endpoint routing (pre-existing codebase issue)
- Making `scripts/sync-skills.ts` resilient to partial GitHub API failures (separate heist)
- Correcting the hardcoded skill count in `sync-skills.ts` (separate heist)
- Any changes to the Nuxt or Docus configuration beyond `wrangler.toml`
